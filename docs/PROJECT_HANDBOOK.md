# Luny Tales v2 — Project Handbook

This document is the technical source of truth for the `lunytales-v2` repository.
It is intended for any engineer who needs to maintain, extend, or migrate the project.

## 1) Project Scope

- Product: Luny Tales website (Spanish + English) built with Astro.
- Deployment target: `https://lunytales.com` (root domain).
- Current architecture: static site generation (`output: "static"`).
- Constraint: do not edit `dist/` manually (build output only).

## 2) Technology Stack

- Runtime: Node.js (CI uses Node 20).
- Framework: Astro `^5.17.1`.
- Styling: global CSS in `public/styles.css` + Bootstrap 5 (CDN).
- Client scripts: vanilla JavaScript in `public/assets/js/`.
- CI/CD: GitHub Actions + `peaceiris/actions-gh-pages`.
- Hosting: GitHub Pages with custom domain (`public/CNAME`).

### Important folders

- `src/pages/`: route definitions.
- `src/layouts/SiteLayout.astro`: shared shell (head, nav, footer, cookie banner, scripts).
- `src/components/HomePage.astro`: shared home markup for ES/EN.
- `src/i18n/`: dictionaries + strict parity validation.
- `src/config/site.ts`: global project configuration (origin, paths, flags).
- `src/lib/seo.ts`: canonical/hreflang/og/twitter generation.
- `public/`: static assets copied as-is to build output.

## 4) Route Map

- `/` → Spanish home.
- `/en/` → English home.
- `/privacy/` → Spanish privacy.
- `/terms/` → Spanish terms.
- `/en/privacy/` → English privacy.
- `/en/terms/` → English terms.
- `/robots.txt` → generated by `src/pages/robots.txt.ts`.
- `/sitemap.xml` → static file from `public/sitemap.xml`.
- `/lunytales-v2/` → redirect shim to `/` (legacy project-page URL support).

## 5) Configuration Model

Single source file: `src/config/site.ts`.

### Current keys

- `IS_STAGING`: derived from `PUBLIC_IS_STAGING === "true"`.
- `SITE_CONFIG.origin`: canonical origin (`https://lunytales.com`).
- `SITE_CONFIG.hotmartUrl`: checkout URL.
- `SITE_CONFIG.paths.*`: route and asset paths.
- `SITE_CONFIG.trackingGate`: controls gated tracking behavior.
- `toHref(path)`: normalizes internal href generation.

### Environment variable

- `PUBLIC_IS_STAGING`
  - `true`: staging SEO behavior (noindex + disallow robots).
  - missing/false: production SEO behavior (index + allow robots).
  - Default is production-safe (false when missing).

## 6) i18n Architecture

Files:

- `src/i18n/es.ts`
- `src/i18n/en.ts`
- `src/i18n/index.ts`

Rules:

- English and Spanish dictionaries must stay in parity.
- Build fails if:
  - any key is missing,
  - unexpected key exists,
  - value types differ,
  - array lengths differ,
  - array object item keysets differ.

This guard is enforced in `assertDictionaryParity()` and runs during imports/build.

## 7) SEO and Indexation

Central generator: `src/lib/seo.ts`.

What it sets:

- `canonical`
- `alternate` hreflang links (`es`, `en`, `x-default`)
- Open Graph tags (`og:title`, `og:description`, `og:url`, `og:image`)
- Twitter card tags
- `robots` meta (`index,follow` or `noindex,nofollow`)

Robots policy:

- Served by `src/pages/robots.txt.ts`.
- Also tied to `IS_STAGING`.
- Includes `Sitemap: https://lunytales.com/sitemap.xml`.

Sitemap:

- Static file `public/sitemap.xml`.
- Must include only live canonical routes unless intentionally expanded.

## 8) Tracking and Consent

Files:

- `public/assets/js/consent.js`
- `public/assets/js/tracking.js`

Behavior:

- Cookie consent is required (`accepted`) before loading tracking.
- If `SITE_CONFIG.trackingGate` is `true`, tracking additionally requires `?track=1`.
- Meta Pixel and custom tracking listeners are loaded dynamically only when allowed.

This prevents accidental full tracking in restricted/staging contexts.

## 9) CI/CD Pipeline

Workflow: `.github/workflows/deploy.yml`

Pipeline steps:

1. Checkout.
2. Install dependencies (`npm ci`).
3. Build (`npm run build`).
4. Verify required build files exist.
5. Run local smoke tests against built output (HTTP 200 checks).
6. Publish `dist/` to `gh-pages` using orphan history.

### Smoke-test coverage (must stay green)

Routes:

- `/`
- `/en/`
- `/privacy/`
- `/terms/`
- `/en/privacy/`
- `/en/terms/`
- `/sitemap.xml`
- `/robots.txt`

Critical assets:

- `/styles.css`
- `/assets/demo/demo.pdf`
- `/assets/demo/demo_eng.pdf`

## 10) Asset Strategy

- Static assets live in `public/assets/...`.
- Refer to assets with project-root-relative paths in templates (compatible with current `<base>` behavior).
- Keep language-specific hero assets separated:
  - ES uses Spanish hero assets.
  - EN uses English hero assets.
- Demo PDFs:
  - ES: `assets/demo/demo.pdf`
  - EN: `assets/demo/demo_eng.pdf`

### Non-negotiable conventions

- Use Conventional Commits.
- Keep commits atomic (one purpose per commit).
- Do not manually edit `dist/`.
- Keep technical docs/comments/commit messages in English.
- Keep config constants centralized (`src/config/site.ts`).
- Do not introduce silent i18n fallback behavior.

| SEO rules/meta generation | `src/lib/seo.ts` |

| Sitemap URLs | `public/sitemap.xml` |
| Custom domain | `public/CNAME` |

## 13) Migration Notes (If Moving Away from GitHub Pages)

If migrating to another host/service:

1. Preserve `SITE_CONFIG.origin` and update it first.
2. Keep canonical/hreflang/robots behavior validated in a staging environment.
3. Replicate current smoke tests (routes + critical assets).
4. Preserve static route behavior and trailing slashes.
5. Keep `/lunytales-v2/` redirect until external links are fully migrated.

Recommended migration sequence:

- Infra first (domain + SSL + caching),
- then deploy pipeline,
- then DNS cutover,
- then post-cutover SEO/tracking validation.

## 14) Operational Checklist Before Release

- `npm run build` passes locally.
- i18n parity check passes.
- Smoke-test routes/assets return 200.
- Canonical and hreflang point to expected production URLs.
- `robots.txt` and `meta robots` match environment intent.
- No manual edits in `dist/`.
- No tokens/secrets committed in repository files.
